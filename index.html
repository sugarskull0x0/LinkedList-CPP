Section 01

Concepts — What Are These Data Structures?

Before we write a single line of code, let's build solid mental models for each structure. A data structure is simply a way of organizing data in memory so we can use it efficiently. The choice of data structure determines how fast we can read, insert, and delete information — so getting this right matters.

Arrays

An array is a fixed-size, ordered collection of elements stored in contiguous (side-by-side) memory locations. When you declare int arr[5]; in C++, the compiler reserves five consecutive int-sized slots in memory. Because they're contiguous, you can jump to any element instantly using its index.

Think of an array like a row of numbered mailboxes at an apartment building. Mailbox 0, Mailbox 1, Mailbox 2 — they sit right next to each other. If you want Mailbox 7, you don't have to walk past 0 through 6; you just go directly to position 7 because you know the exact spacing.

Singly Linked List

A singly linked list is a chain of nodes where each node contains two things: the data it stores, and a pointer to the next node in the chain. The first node is called the head. The last node's "next" pointer is nullptr, signaling the end of the list. Nodes can live anywhere in memory — they don't need to be adjacent.

Doubly Linked List

A doubly linked list is the same idea, but each node has two pointers: one pointing to the next node and one pointing to the previous node. This lets you traverse the list in both directions. You typically keep track of both the head (first node) and the tail (last node).

Visual: Array vs Singly vs Doubly Linked List
Key Takeaway

An array stores elements contiguously (side-by-side) and uses index-based access. A singly linked list chains nodes with forward-only pointers. A doubly linked list adds backward pointers for bidirectional traversal. The fundamental trade-off is between random access speed (arrays win) and insertion/deletion flexibility (linked lists win).

Section 02

Array Pros & Cons (Memory Behavior)

When you write int arr[5];, the compiler allocates 5 × 4 = 20 bytes of contiguous memory. "Contiguous" means every byte is directly adjacent — no gaps. This has profound performance implications.

Why Contiguous Memory Matters

Modern CPUs don't fetch one byte at a time from RAM — they pull entire cache lines (usually 64 bytes). When you access arr[0], the CPU loads arr[0] through arr[15] (or more) into the cache in one shot. When you then access arr[1], it's already in the cache — a cache hit. This is called spatial locality, and it makes sequential array traversals blazingly fast.

Memory Layout: Array of 5 Integers

Random access is O(1) because the address of any element can be calculated with a simple formula: address = base_address + (index × sizeof(element)). To find arr[3], the CPU computes 0x1000 + (3 × 4) = 0x100C and jumps straight there.

Contiguous memory is like a row of seats in a cinema. If you know the row starts at Seat 1, finding Seat 47 is trivial — you don't scan every seat, you just count 46 seats over. But if you need to squeeze someone into the middle of a full row, everyone past that point has to shift over one seat.
Advantages
  • O(1) random access — jump to any element by index instantly
  • Cache-friendly — sequential access benefits from hardware prefetching
  • Zero pointer overhead — all memory goes to actual data
  • Simple syntax — easy to declare, read, and use
Disadvantages
  • Fixed size — C-style arrays can't grow after declaration
  • Expensive insertion/deletion — O(n) shifting of elements
  • Wasteful if sparse — allocated memory is used even if slots are empty
  • Fragmentation risk — large contiguous blocks may fail to allocate
Key Takeaway

Arrays are the fastest structure for reading data by position thanks to contiguous memory and O(1) indexing. They struggle with insertions and deletions in the middle because every element after the change must shift. Choose arrays when your data has a predictable size and you need frequent, fast lookups.

Section 03

Linked List Pros & Cons (Memory Behavior)

Linked list nodes are allocated individually with new, so they can live anywhere in memory — address 0x2000, 0x5FA0, 0x8120 — completely scattered. Each node carries its data plus one or two pointers to connect itself to the chain.

Memory Layout: Singly Linked List (3 Nodes, Scattered)
A linked list is like a scavenger hunt. Each clue (node) tells you where to find the next clue — but the clues are hidden all over the building. You can't skip ahead; you must follow each clue to the next. With a doubly linked list, each clue also tells you where the previous one was, so you can retrace your steps.

Singly Linked List Trade-offs

With only a forward pointer, you can only traverse head-to-tail. Inserting or deleting at the head is O(1) — just rewire one pointer. But inserting at position k requires walking k steps first to find the right spot, making it O(n) for the search plus O(1) for the actual re-linking.

Doubly Linked List Trade-offs

The extra prev pointer doubles the pointer overhead (now 16 bytes per node on a 64-bit system), but gives you two superpowers: you can delete a node in O(1) if you already have a pointer to it (because you can reach both neighbors), and you can traverse backwards from the tail. Inserting/deleting at both head and tail becomes O(1).

Advantages
  • Dynamic size — grows and shrinks freely at runtime
  • O(1) insertion/deletion at known positions (no shifting)
  • No wasted space — allocates exactly as many nodes as needed
  • No large contiguous block needed — uses scattered memory
Disadvantages
  • No random access — O(n) to reach element k
  • Pointer overhead — 8–16 extra bytes per node
  • Poor cache locality — scattered nodes cause cache misses
  • Memory fragmentation — many small allocations stress the allocator
Key Takeaway

Linked lists trade access speed for modification flexibility. They shine when you insert and delete frequently, especially at the ends. The doubly linked variant is more versatile but uses more memory. Both suffer from poor cache performance compared to arrays because nodes are scattered in RAM.

Section 04

Code Examples — Arrays in C++

Let's see arrays in action. We'll cover creation, access, insertion (shifting), and deletion (shifting). We'll use C-style arrays first, then show how std::vector handles dynamic sizing.

arrays_demo.cpp 
// ============================================================
// ARRAYS IN C++ — Creation, Access, Insertion, Deletion
// ============================================================
#include <iostream>
using namespace std;

// ---- Helper: Print an array ----
void printArray(int arr[], int size) {
    cout << "[ ";
    for (int i = 0; i < size; i++) {
        cout << arr[i];
        if (i < size - 1) cout << ", ";
    }
    cout << " ]" << endl;
}

// ---- Insert a value at a given index (shifts elements right) ----
// 'size' is the current number of elements; 'capacity' is max.
void insertAt(int arr[], int& size, int capacity, int index, int value) {
    if (size >= capacity) {
        cout << "Error: array is full!" << endl;
        return;
    }
    if (index < 0 || index > size) {
        cout << "Error: index out of bounds!" << endl;
        return;
    }
    // Shift everything from 'index' onward one position to the right.
    // This is the expensive O(n) part.
    for (int i = size; i > index; i--) {
        arr[i] = arr[i - 1];
    }
    arr[index] = value;
    size++;
}

// ---- Delete the element at a given index (shifts elements left) ----
void deleteAt(int arr[], int& size, int index) {
    if (index < 0 || index >= size) {
        cout << "Error: index out of bounds!" << endl;
        return;
    }
    // Shift everything after 'index' one position to the left.
    for (int i = index; i < size - 1; i++) {
        arr[i] = arr[i + 1];
    }
    size--;
}

int main() {
    // --- 1. CREATION ---
    const int CAPACITY = 10;    // Maximum slots available
    int arr[CAPACITY] = {10, 20, 30, 40, 50};
    int size = 5;              // Currently holding 5 elements

    cout << "Initial array: ";
    printArray(arr, size);      // [ 10, 20, 30, 40, 50 ]

    // --- 2. ACCESS (O(1) — direct index) ---
    cout << "Element at index 2: " << arr[2] << endl;   // 30

    // --- 3. INSERTION at index 2 (O(n) — must shift) ---
    insertAt(arr, size, CAPACITY, 2, 25);
    cout << "After inserting 25 at index 2: ";
    printArray(arr, size);      // [ 10, 20, 25, 30, 40, 50 ]

    // --- 4. DELETION at index 0 (O(n) — must shift) ---
    deleteAt(arr, size, 0);
    cout << "After deleting index 0: ";
    printArray(arr, size);      // [ 20, 25, 30, 40, 50 ]

    return 0;
}

Notice the pattern: accessing is instant, but inserting and deleting require a loop to shift elements. That loop is what gives arrays their O(n) insertion/deletion cost.

Key Takeaway

Array code is straightforward but you must manually manage shifting for insertions and deletions. In practice, std::vector automates this (it's a dynamic array under the hood), but the O(n) shifting cost remains. Understanding the raw mechanics helps you reason about when arrays are the right choice.

Section 05

Code Implementation — Singly Linked List

Here's a complete, working singly linked list with insert (head, tail, at position), delete, search, and traversal. Each operation is commented to explain the pointer manipulation.

singly_linked_list.cpp 
// ============================================================
// SINGLY LINKED LIST — Complete C++ Implementation
// ============================================================
#include <iostream>
using namespace std;

// ---- Node structure: data + one pointer ----
struct Node {
    int   data;
    Node* next;

    // Constructor for convenient allocation
    Node(int val) : data(val), next(nullptr) {}
};

// ---- Singly Linked List class ----
class SinglyLinkedList {
private:
    Node* head;

public:
    SinglyLinkedList() : head(nullptr) {}

    // ---- INSERT AT HEAD — O(1) ----
    // Create a new node, point it to the old head, update head.
    void insertAtHead(int val) {
        Node* newNode = new Node(val);
        newNode->next = head;   // New node points to old head
        head = newNode;          // Head now points to new node
    }

    // ---- INSERT AT TAIL — O(n) ----
    // Must traverse the entire list to find the last node.
    void insertAtTail(int val) {
        Node* newNode = new Node(val);
        if (!head) {             // Empty list: new node becomes head
            head = newNode;
            return;
        }
        Node* curr = head;
        while (curr->next) {     // Walk to the last node
            curr = curr->next;
        }
        curr->next = newNode;    // Last node now points to new node
    }

    // ---- INSERT AT POSITION — O(n) ----
    // Position 0 = head. Walk to position-1, then re-link.
    void insertAtPosition(int val, int pos) {
        if (pos == 0) { insertAtHead(val); return; }

        Node* curr = head;
        for (int i = 0; i < pos - 1 && curr; i++) {
            curr = curr->next;   // Walk to the node BEFORE target position
        }
        if (!curr) {
            cout << "Position out of range!" << endl;
            return;
        }
        Node* newNode = new Node(val);
        newNode->next = curr->next;  // New node points to what came after
        curr->next = newNode;        // Previous node now points to new node
    }

    // ---- DELETE BY VALUE — O(n) ----
    // Find the node with matching data, re-link, then free memory.
    void deleteByValue(int val) {
        if (!head) return;

        // Special case: deleting the head
        if (head->data == val) {
            Node* temp = head;
            head = head->next;   // Move head forward
            delete temp;          // Free old head's memory
            return;
        }

        // General case: find node BEFORE the one to delete
        Node* curr = head;
        while (curr->next && curr->next->data != val) {
            curr = curr->next;
        }
        if (curr->next) {
            Node* temp = curr->next;
            curr->next = temp->next;  // Skip over the deleted node
            delete temp;              // Free its memory
        }
    }

    // ---- SEARCH — O(n) ----
    bool search(int val) {
        Node* curr = head;
        while (curr) {
            if (curr->data == val) return true;
            curr = curr->next;
        }
        return false;
    }

    // ---- DISPLAY — O(n) ----
    void display() {
        Node* curr = head;
        while (curr) {
            cout << curr->data << " -> ";
            curr = curr->next;
        }
        cout << "NULL" << endl;
    }

    // ---- DESTRUCTOR — free all nodes ----
    ~SinglyLinkedList() {
        while (head) {
            Node* temp = head;
            head = head->next;
            delete temp;
        }
    }
};

int main() {
    SinglyLinkedList list;

    list.insertAtTail(10);
    list.insertAtTail(20);
    list.insertAtTail(30);
    list.display();                // 10 -> 20 -> 30 -> NULL

    list.insertAtHead(5);
    list.display();                // 5 -> 10 -> 20 -> 30 -> NULL

    list.insertAtPosition(15, 2);
    list.display();                // 5 -> 10 -> 15 -> 20 -> 30 -> NULL

    list.deleteByValue(20);
    list.display();                // 5 -> 10 -> 15 -> 30 -> NULL

    cout << "Search 15: " << (list.search(15) ? "Found" : "Not found") << endl;
    cout << "Search 99: " << (list.search(99) ? "Found" : "Not found") << endl;

    return 0;
}
Key Takeaway

The core skill is pointer rewiring: to insert, you redirect the previous node's next to the new node, and the new node's next to what came after. To delete, you skip over the target node and delete it. Always handle the head-node special case separately, and always free memory with delete to prevent leaks.

Section 06

Code Implementation — Doubly Linked List

The doubly linked list adds a prev pointer, making deletion at a known node O(1) and enabling backward traversal. The trade-off: more pointers to manage on every operation.

doubly_linked_list.cpp 
// ============================================================
// DOUBLY LINKED LIST — Complete C++ Implementation
// ============================================================
#include <iostream>
using namespace std;

// ---- Node structure: data + two pointers ----
struct DNode {
    int    data;
    DNode* prev;
    DNode* next;

    DNode(int val) : data(val), prev(nullptr), next(nullptr) {}
};

// ---- Doubly Linked List class ----
class DoublyLinkedList {
private:
    DNode* head;
    DNode* tail;

public:
    DoublyLinkedList() : head(nullptr), tail(nullptr) {}

    // ---- INSERT AT HEAD — O(1) ----
    void insertAtHead(int val) {
        DNode* newNode = new DNode(val);
        if (!head) {
            head = tail = newNode;   // First node: it's both head and tail
            return;
        }
        newNode->next = head;     // New node's next → old head
        head->prev = newNode;     // Old head's prev → new node
        head = newNode;           // Update head pointer
    }

    // ---- INSERT AT TAIL — O(1) ----
    // Unlike the singly list, this is O(1) because we track the tail!
    void insertAtTail(int val) {
        DNode* newNode = new DNode(val);
        if (!tail) {
            head = tail = newNode;
            return;
        }
        newNode->prev = tail;     // New node's prev → old tail
        tail->next = newNode;     // Old tail's next → new node
        tail = newNode;           // Update tail pointer
    }

    // ---- INSERT AT POSITION — O(n) ----
    void insertAtPosition(int val, int pos) {
        if (pos == 0) { insertAtHead(val); return; }

        DNode* curr = head;
        for (int i = 0; i < pos && curr; i++) {
            curr = curr->next;
        }
        if (!curr) {     // pos is at or past end → insert at tail
            insertAtTail(val);
            return;
        }
        // Insert BEFORE curr
        DNode* newNode = new DNode(val);
        newNode->prev = curr->prev;
        newNode->next = curr;
        curr->prev->next = newNode;
        curr->prev = newNode;
    }

    // ---- DELETE BY VALUE — O(n) search + O(1) removal ----
    void deleteByValue(int val) {
        DNode* curr = head;

        // Search for the node
        while (curr && curr->data != val) {
            curr = curr->next;
        }
        if (!curr) return;    // Value not found

        // Re-link neighbors around the doomed node
        if (curr->prev) curr->prev->next = curr->next;
        else            head = curr->next;  // Deleting head

        if (curr->next) curr->next->prev = curr->prev;
        else            tail = curr->prev;  // Deleting tail

        delete curr;
    }

    // ---- DISPLAY FORWARD — O(n) ----
    void displayForward() {
        DNode* curr = head;
        while (curr) {
            cout << curr->data << " <-> ";
            curr = curr->next;
        }
        cout << "NULL" << endl;
    }

    // ---- DISPLAY BACKWARD — O(n) ----
    // This is a superpower of doubly linked lists!
    void displayBackward() {
        DNode* curr = tail;
        while (curr) {
            cout << curr->data << " <-> ";
            curr = curr->prev;
        }
        cout << "NULL" << endl;
    }

    // ---- DESTRUCTOR ----
    ~DoublyLinkedList() {
        while (head) {
            DNode* temp = head;
            head = head->next;
            delete temp;
        }
    }
};

int main() {
    DoublyLinkedList dll;

    dll.insertAtTail(10);
    dll.insertAtTail(20);
    dll.insertAtTail(30);
    dll.insertAtHead(5);
    dll.insertAtPosition(15, 2);

    cout << "Forward:  ";
    dll.displayForward();      // 5 <-> 10 <-> 15 <-> 20 <-> 30 <-> NULL

    cout << "Backward: ";
    dll.displayBackward();     // 30 <-> 20 <-> 15 <-> 10 <-> 5 <-> NULL

    dll.deleteByValue(15);
    dll.deleteByValue(5);      // Delete head
    dll.deleteByValue(30);     // Delete tail

    cout << "After deletions: ";
    dll.displayForward();      // 10 <-> 20 <-> NULL

    return 0;
}
Key Takeaway

Doubly linked list operations require updating two pointers per node instead of one — but the reward is O(1) deletion when you have a direct reference to the node, O(1) tail insertion, and backward traversal. The deletion code elegantly handles all edge cases (head, tail, middle) by checking prev and next before relinking.

Big-O Comparison Table

Operation Array Singly Linked Doubly Linked
Access by index O(1) O(n) O(n)
Search by value O(n) O(n) O(n)
Insert at beginning O(n) O(1) O(1)
Insert at end O(1)* O(n) O(1)
Insert at middle O(n) O(n) O(n)
Delete at beginning O(n) O(1) O(1)
Delete at end O(1)* O(n) O(1)
Delete known node O(n) O(n) O(1)
Memory per element Just the data Data + 1 pointer Data + 2 pointers
Cache performance Excellent Poor Poor

* Array O(1) end operations assume there's room and you track the current size. std::vector provides amortized O(1) push_back with occasional O(n) reallocation.

Section 08

Real-World Use Cases

Choosing the right data structure isn't academic — it directly impacts how fast and memory-efficient your software is. Here's when to reach for each one in practice.

Arrays
  • Image pixel buffers (row-major grid access)
  • Lookup tables and hash maps (index-based)
  • Audio sample buffers (sequential playback)
  • Matrix math and scientific computing
  • Any fixed-size collection read more than modified
Singly Linked
  • Stack implementation (push/pop at head)
  • Chained hash table buckets
  • Simple task queues (FIFO with tail pointer)
  • Polynomial representation in math software
  • Memory-constrained embedded systems
Doubly Linked
  • Browser back/forward navigation
  • Undo/redo systems in editors
  • LRU cache eviction (fast delete + move)
  • Music playlist with prev/next track
  • OS process scheduler (round-robin queues)

How to Decide

Default to arrays (or std::vector) unless you have a specific reason not to. Arrays are simpler, more cache-friendly, and faster for most workloads. Reach for a singly linked list when you need frequent insertions/deletions at the front and don't need backward traversal or random access — it uses less memory than its doubly linked sibling. Choose a doubly linked list when you need bidirectional traversal, or when you'll be deleting nodes given a direct pointer to them (like in an LRU cache where you keep a map of key → node pointer).

Here's a rough rule of thumb: if you're building something that reads a lot and changes rarely, use an array. If it changes constantly at the ends, use a linked list. If it changes constantly and you need to navigate both directions, use a doubly linked list.
Final Takeaway

There is no universally "best" data structure — only the best one for your specific access pattern. Understanding how arrays, singly linked lists, and doubly linked lists behave in memory is the foundation that lets you make that choice confidently. When in doubt, profile your code. The theory predicts big-O trends, but real-world cache behavior often makes arrays even faster than you'd expect.
